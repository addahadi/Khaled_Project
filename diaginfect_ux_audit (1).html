Section 1.1

Authentication — Login, Register, Activate

Login Page src/pages/auth/Login.tsx
Strong Ergonomic Implementations
✓ Client-side Zod validation fires before the API call, providing immediate inline error messages per field — satisfies Bastien & Scapin's "Error Management" criterion.
✓ autoComplete="email" and autoComplete="current-password" are correctly set, leveraging browser password managers and reducing manual entry burden.
✓ The submit button is disabled with a spinner during the API call — clear system status feedback (Nielsen Heuristic #1).
✓ Role-aware post-login redirect via DASHBOARD map prevents users from ever landing on the wrong portal.
Issues & Better Approaches
✕ No password visibility toggle. The type="password" field provides no eye icon. Clinical users often type complex passwords; masking with no reveal option increases login failure rate.
→ Better Approach
Add an <Eye/EyeOff> icon button inside the input's right padding (using absolute positioning) that toggles type between "password" and "text". Mark it aria-label="Show password".
✕ No "Forgot Password" link. This is a critical missing affordance. Clinical staff who rotate passwords frequently will hit this wall.
→ Better Approach
Add a "Forgot password?" link under the password field, right-aligned, that navigates to a /forgot-password flow. This is the most-expected pattern per iOS HIG & Material Design guidelines.
✕ Generic Activity icon as app logo. The branding uses a generic activity/heartbeat icon that carries no clinical-diagnostic meaning. For a disease-diagnosis platform, this creates weak brand recall.
→ Better Approach
Use a compound icon (e.g., a microscope + brain outline or a stylized pathogen) that communicates both laboratory and AI aspects. Alternatively, a text wordmark in a medical typeface creates stronger recall.
Register Organization — Multi-Step Wizard src/pages/auth/RegisterOrganization.tsx
Strong Ergonomic Implementations
✓ Step indicator with completed state (green checkmark), active state (primary colour), and inactive state (muted) provides strong orientation — excellent application of Bastien & Scapin's "Guidance" criterion.
✓ Per-step Zod validation prevents users from progressing with invalid data, enforcing progressive disclosure correctly.
✓ Plans are lazily loaded only when step 3 is reached — efficient use of network resources.
Issues & Better Approaches
✕ Step labels hidden on mobile via hidden sm:block. On a 375px phone the step indicator shows only numbered circles with no textual context. A user at step 2 has no hint that the next step is "Choose Plan."
→ Better Approach
Below the numbered circles, show the current step label always: "Step 2 of 3 — Organization Details". This single line costs 20px of height and entirely resolves mobile orientation loss.
✕ No password strength meter. The password field in Step 1 accepts any string that passes Zod's minimum length. Clinical platform accounts may hold PHI — a strength indicator is both a security and trust signal.
→ Better Approach
Add a 4-segment progress bar beneath the password field with labels: Weak / Fair / Strong / Very Strong, driven by entropy estimation (e.g., zxcvbn library). Show it only after the first keystroke.
✕ Plan selection has no loading skeleton for step 3. Plans are fetched asynchronously when entering step 3, but there is no loading state; the card area is blank until data arrives.
→ Better Approach
Render 3 <Skeleton> plan-card placeholders while plans.length === 0 and the fetch is in flight, identical to the skeleton patterns used elsewhere in the app for consistency.
Section 1.2

Doctor — Dashboard

Doctor Dashboard src/pages/doctor/Dashboard.tsx + DoctorLayout.tsx
Strong Ergonomic Implementations
✓ 4-card KPI grid with individual loading skeletons — each stat loads independently without blocking the whole view. Excellent granularity.
✓ RISK_STYLE color map is applied consistently across the dashboard, patient cards, and prediction rows — strong visual consistency (ISO 9241-11: Consistency).
✓ The empty-state for no patients renders a contextual CTA ("Register one") rather than a dead-end message — excellent error recovery guidance.
✓ Unread alert count badge on the sidebar-Bell and in the Alerts section title gives dual-channel notification.
Issues & Better Approaches
✕ CRITICAL — Bell notification button navigates to /doctor/dashboard, not to Alerts. In DoctorLayout.tsx: onClick={() => navigate('/doctor/dashboard')}. A user on the Patients page sees 5 unread alerts, clicks the bell, and is taken to… the same dashboard, not to an alerts detail view. This violates the "Match between System and Real World" heuristic — a bell universally means "show me my notifications."
→ Better Approach
Navigate to /doctor/alerts (create a dedicated alerts page) or anchor to the alerts section: navigate('/doctor/dashboard', { state: { scrollTo: 'alerts' } }). The dedicated page is strongly preferred for scalability.
✕ CRITICAL — "Mark as read" button is 16×16px (h-4 w-4). The blue dot button that marks an alert as read is a standalone interactive element of only 16×16px. WCAG 2.5.5 requires a minimum 44×44px touch target. On a tablet this is nearly impossible to tap accurately in a clinical environment.
→ Better Approach
Replace the dot with a labeled button: <Button variant="ghost" size="sm" onClick={() => markRead(a.alert_id)}>Mark read</Button>. This gives a 36–44px touch target and is self-describing.
✕ Duplicate API calls on every layout mount. Both DoctorSidebar and DoctorLayout independently call /doctor/alerts via ApiManager.execute on mount. This fires two identical requests every time the layout renders.
→ Better Approach
Lift the alerts state to a shared context (e.g., extend AuthContext or create an AlertsContext) so the count is fetched once, shared across both consumers, and updated reactively when markRead is called.
✕ Prediction cards navigate to the general list, not the detail. Clicking a recent prediction card calls navigate('/doctor/predictions') — it drops the user at the list, requiring an additional click to find the specific item.
→ Better Approach
Navigate directly to the specific prediction: navigate(\`/doctor/predictions/${pr.request_id}\`). The PredictionHistory component already has an openDetail mechanism for this.
Section 1.3

Doctor — Patients List

Patients List src/pages/doctor/Patients.tsx
Strong Ergonomic Implementations
✓ Dual-mode filtering: text search + risk-level toggle buttons. Both work simultaneously and reset independently — excellent flexibility.
✓ Three empty-state variations (loading, no results for search, genuinely empty list) each with contextual CTAs. No dead-end states.
✓ Risk score percentage shown alongside the badge gives clinicians a quantitative anchor alongside the categorical label.
Issues & Better Approaches
✕ No column sorting. The patient grid has no sort-by-name, sort-by-risk, or sort-by-date capability. A doctor with 200 patients who wants to triage the most recently added cannot do so.
→ Better Approach
Add a sort dropdown (or heading-click-to-sort if switching to a list layout): "Sort by: Name A–Z | Risk: Highest First | Recently Added". Client-side sort on the patients array before rendering filtered.
✕ Risk classification is color-coded only. CRITICAL, HIGH, MODERATE, LOW are differentiated only by background color. For users with deuteranopia (red-green color blindness), CRITICAL red and MODERATE amber look identical.
→ Better Approach
Add a unique prefix icon to each risk badge: CRITICAL → <AlertTriangle>, HIGH → <TrendingUp>, MODERATE → <Minus>, LOW → <CheckCircle2>. The icon provides a redundant visual cue that is color-agnostic.
✕ Patient registration form collects only Name, Age, Gender. The medical_history field exists in the backend schema and patient detail view, but registration never populates it. Doctors must navigate back to add this data separately.
→ Better Approach
Add an optional "Medical History" text area (or tag-input for conditions) to the registration dialog under a "Advanced (optional)" collapsible section. Pre-filling this at registration reduces the number of follow-up steps.
Section 1.4

Doctor — Patient Detail

Patient Detail src/pages/doctor/PatientDetail.tsx
Strong Ergonomic Implementations
✓ Tab-based layout (Overview / Clinical / Lab / Predictions) cleanly partitions high-density data without nesting. Badge counts in tab labels aid rapid triage.
✓ Risk banner with left color-border (border-l-4) is prominent and non-intrusive — it's visible without dominating the page.
✓ Edit and delete actions on clinical records are contextual (per-card), not global — limiting the blast radius of accidental clicks.
✓ Lab order dialog pre-focuses on the test-type field by default (first focusable element), enabling fast keyboard-first entry.
Issues & Better Approaches
✕ CRITICAL — Symptoms field uses a raw comma-separated text input. Clinical symptoms are unstructured free text: placeholder="fever, cough, fatigue". Users can enter "Fever", "fever", "FEVER" — all as separate values. This pollutes AI training data and makes symptom trends impossible to track.
→ Better Approach
Replace with a tag-input component backed by a curated symptom list (combobox). Show autocomplete suggestions from a predefined clinical vocabulary (ICD-10 symptoms, or a curated list of 50–100 common infectious disease symptoms). Still allow free-text additions as a fallback.
✕ CRITICAL — Deleting clinical data has no upstream impact warning. The confirmation dialog reads "This action cannot be undone" but says nothing about the fact that deleting vitals data will affect the quality (or possibility) of future AI predictions for this patient.
→ Better Approach
Extend the confirmation dialog: "Deleting this record will remove it from AI prediction inputs. Any future predictions for [Patient Name] will no longer include this clinical snapshot. This cannot be undone." This satisfies ISO 9241-110's "Error Tolerance" principle.
✕ Lab test type is a free-text field. placeholder="e.g. CBC, Blood Culture, Urinalysis". A doctor might type "cbc", "CBC", "Complete Blood Count" — all referring to the same test. Lab technicians then see inconsistent test names.
→ Better Approach
Replace with a <Select> or combobox populated with standardized test names from a /lab/test-types endpoint. Allow custom entry only as a last resort, with a warning: "Custom test types may not be recognized by the AI prediction model."
✕ No breadcrumb navigation. On mobile, the back-arrow returns to /doctor/patients only. If the user arrived from Dashboard's "recent patients" list, the back arrow still goes to the patients list, not the dashboard. There's no breadcrumb like "Dashboard → Patients → Ahmed Benali."
→ Better Approach
Add a breadcrumb bar: Patients / {patient.name}. The "Patients" segment links to the list. This costs one line of JSX and solves navigation orientation completely.
Section 1.5

Doctor — New Prediction

New AI Prediction src/pages/doctor/NewPrediction.tsx
Strong Ergonomic Implementations
✓ The "Data used for prediction" summary card contextualizes the AI's inputs before the doctor submits — builds trust and transparency (XAI principle).
✓ Risk result card uses dynamic colour border + background tinting (e.g., border-red-300 bg-red-50) that scales from LOW to CRITICAL — excellent progressive urgency feedback.
✓ Overage warning is surfaced both as a badge and as an inline message box — dual-channel notification for a billing-sensitive event.
Issues & Better Approaches
✕ CRITICAL — Developer annotation leaked into production UI. The page subtitle reads: "Image 2 — runs Random Forest / XGBoost / NN on clinical + lab data". This is a developer comment that was never removed. It directly violates Nielsen's Aesthetic and Minimalist Design heuristic and destroys clinical credibility.
→ Better Approach
Replace with a clinically meaningful subtitle: "Runs an ensemble AI model on the patient's latest clinical record and completed lab results." Audit all pages for similar inline annotations.
✕ Model Version selector exposes engineering internals to clinicians. A doctor has no clinical basis for choosing between v2.3.1, v2.2.0, and v2.1.5. Showing version numbers creates decision paralysis and forces meaningless choices.
→ Better Approach
Default to the latest model silently. If version selection is required for audit trails, remove it from the doctor's UI and let it be a manager/admin setting. If doctors must see it, label it as "Recommended (latest)" / "Previous version" with a changelog tooltip.
✕ "Confidence" percentage is unexplained for clinicians. The result card shows "Confidence: 87%" with no tooltip explaining what this means. A doctor might interpret this as "87% certainty the patient has an infection" rather than its actual meaning (model confidence in its own classification).
→ Better Approach
Add an info icon (InfoCircle) next to the Confidence label with a tooltip: "Model confidence reflects how certain the AI is about its risk classification, based on the completeness and consistency of input data. It does not represent the probability of infection."
Section 1.6

Doctor — Prediction History

Prediction History + XAI Sheet src/pages/doctor/PredictionHistory.tsx
Strong Ergonomic Implementations
✓ Summary stat tiles at the top are clickable filters — tapping "3 CRITICAL" immediately filters the list to CRITICAL only. Excellent direct manipulation.
✓ XAI feature explanations are surfaced in a side-sheet rather than a navigation away, preserving context and enabling quick review-and-return.
✓ Feature direction (↑ increases risk / ↓ decreases risk) uses both an arrow symbol AND red/green color coding — redundant encoding that is partially color-blind safe.
Issues & Better Approaches
✕ XAI feature names are raw backend identifiers. Features like temperature, heart_rate, wbc_count appear as-is. While these are readable, more complex names from ML feature engineering (e.g., crp_log_transformed, neutrophil_lymphocyte_ratio) would be opaque to most clinicians.
→ Better Approach
Maintain a frontend label map (similar to the existing FEATURE_LABELS in Subscription.tsx) that translates feature identifiers to human-readable clinical names. Example: "neutrophil_lymphocyte_ratio" → "Neutrophil-to-Lymphocyte Ratio (NLR)". Add brief clinical context on hover.
✕ Contribution bars are 1.5px tall (h-1.5). The XAI bar charts are 6px tall and 96px wide. They are functionally invisible at arm's length and inaccessible to screen readers (no ARIA roles, no text alternative for the bar value).
→ Better Approach
Increase bars to at least h-3 (12px). Add role="meter", aria-valuenow, aria-valuemin="0", aria-valuemax="100" attributes. Consider using a horizontal bar label with the percentage text always visible rather than only on hover.
✕ No date-range filter. Doctors reviewing trends cannot filter predictions to "this week" or "last 30 days." The only filter is by risk level.
→ Better Approach
Add a date-range picker (from/to or quick selections: "Today / Last 7 Days / This Month / All Time") that filters the predictions array client-side. This is a high-frequency clinical workflow for progress monitoring.
Section 1.7

Lab — Dashboard & Orders

Lab Dashboard + Lab Orders src/pages/lab/LabDashboard.tsx + LabOrders.tsx
Strong Ergonomic Implementations
✓ Dashboard shows "Completed Today" and "Total Completed" as separate stats — reflects the lab tech's actual daily mental model (shift-based work).
✓ Inline Start → Enter Results action flow in the dashboard table eliminates the need to navigate to a separate page for workflow initiation.
✓ Search in LabOrders.tsx searches by both patient name AND test type — covering both primary lookup methods a lab tech uses.
Issues & Better Approaches
✕ CRITICAL — PENDING orders show both "Start" AND "Enter Results" simultaneously in LabOrders.tsx. The condition (o.status === 'INPROGRESS' || o.status === 'PENDING') renders "Enter Results" for PENDING orders, while a separate condition also renders "Start". A PENDING order shows two competing CTAs with no guidance on which to use first.
→ Better Approach
Apply a strict state machine: PENDING → only show "Start". INPROGRESS → only show "Enter Results". Remove the OR condition. Optionally, combine them as a single "Begin Test" button that starts AND navigates to entry in one action.
✕ No urgency/age indicators on pending orders. The orders table shows the order date but no indication of how long it has been pending. A 3-hour-old STAT order looks identical to a routine order placed yesterday.
→ Better Approach
Add a relative time column ("2h ago / 3 days ago") using a library like date-fns/formatDistanceToNow. Highlight orders older than N hours with an amber badge "Overdue". Sort PENDING by age ascending by default (oldest first = most urgent).
Section 1.8

Lab — Enter Results

Enter Lab Results src/pages/lab/EnterResults.tsx
Strong Ergonomic Implementations
✓ Auto-computed flag ("Server-computed") with a blue "auto" label is a superb UX pattern — it reduces manual work and prevents human error in classification while remaining transparent.
✓ Completed orders are rendered in full disabled read-only state rather than a separate page — lab techs can review results in context without extra navigation.
✓ Row removal is protected: "Remove" button only appears when rows.length > 1, preventing the user from accidentally emptying all entries.
Issues & Better Approaches
✕ No test templates or presets. For a CBC, a lab tech must manually add 8–15 analyte rows one by one. There are no templates. This is the highest-friction workflow in the entire application for the lab role.
→ Better Approach
Add a "Load Template" button that pre-populates rows with standard analytes for the test type. Templates could be managed by the manager, or built-in for common tests (CBC: WBC, RBC, HGB, HCT, PLT, MCV, MCH, MCHC). This single feature could reduce lab entry time by 60–80%.
✕ "Panel Group" label is technical jargon. The sub_panel field is labeled "Panel Group" with a placeholder "e.g. CBC_DIFF". The underscore-separated all-caps format is an internal database convention, not clinical vocabulary.
→ Better Approach
Label it "Test Group (optional)" with placeholder "e.g. Differential, Electrolytes". Use human-readable values that map to the backend's expected format, or accept any grouping string and let the server normalize it.
✕ No in-row validation feedback during entry. Validation errors appear as a toast notification plus inline errors, but only after clicking Submit. A lab tech entering 12 analytes won't discover a missing field until they've completed everything.
→ Better Approach
Apply per-row onBlur validation using the Zod row schema. Mark each row's required fields with a red border immediately on blur. This provides instant feedback per field rather than a batch error at submission.

Manager — Dashboard & Staff

Manager Dashboard + Staff Management src/pages/manager/ManagerDashboard.tsx + Staff.tsx
Strong Ergonomic Implementations
✓ Risk distribution bars on the dashboard update dynamically from real prediction data, giving managers a live quality-of-care signal.
✓ Usage progress bar turns destructive red at 85% threshold — proactive notification before hitting the limit, not after.
✓ Staff table provides role-colour badges AND status-colour badges simultaneously — allows scanning for "active doctors" vs "suspended lab techs" at a glance.
Issues & Better Approaches
✕ CRITICAL — Status changes (Suspend/Deactivate) fire immediately with no confirmation dialog. The dropdown onClick={() => updateStatus(m.user_id, 'SUSPENDED')} patches the API instantly. A manager misclicking "Suspend" instead of "Deactivate" on a doctor actively using the system creates an immediate access disruption.
→ Better Approach
Intercept destructive actions (INACTIVE, SUSPENDED) with a confirmation dialog: "Are you sure you want to suspend [Dr. Name]? They will immediately lose access to all patient records. This can be reversed from this page." Keep "Activate" as a direct action since it's restorative, not destructive.
✕ No reason/note field for suspension. When a manager suspends a staff member, no audit note is recorded. HR and compliance requirements in healthcare typically mandate that account suspensions carry a reason code.
→ Better Approach
Add an optional "Reason" textarea to the suspension confirmation dialog. Submit as { status: 'SUSPENDED', note: '...' }. Display the note in a tooltip on the staff row's status badge for audit traceability.
✕ Risk distribution bars lack ARIA roles. The custom progress bars in the Manager Dashboard are plain <div> elements with no ARIA attributes. Screen readers will announce them as anonymous content with no semantic meaning.
→ Better Approach
Add role="meter", aria-label="Critical Risk: 5 predictions", aria-valuenow={5}, aria-valuemin={0}, aria-valuemax={total} to each bar's container. Alternatively, switch to the shadcn <Progress> component which includes these by default.

Section 2

Missing Ergonomic Criteria — Critical & High Priority

🚫 Session Timeout Warning
No session expiry warning exists anywhere in the application. JWT tokens will expire silently. A doctor mid-way through entering clinical data will have their form state destroyed when the session expires, with no warning and no save prompt. In a PHI context, this is both a workflow and a compliance failure.
→ Implement a countdown modal at T-5 minutes before token expiry offering "Stay logged in" or "Log out securely". Use a silent token-refresh call to extend on confirmation.
🚫 Keyboard Navigation & Focus Management
Modal dialogs (Add Patient, Lab Order, Delete Confirmation) do not implement focus trapping. When a dialog opens, keyboard focus is not moved into it. When it closes, focus is not returned to the triggering element. This breaks the WCAG 2.1 criterion 2.4.3 (Focus Order) and renders the app non-functional for keyboard-only users.
→ Use shadcn's Dialog component's built-in focus management (which uses Radix UI), verify it's not being overridden. Add initialFocusRef to direct focus to the primary input on open.
🚫 No "Forgot Password" Flow
The authentication system has no password reset mechanism. For clinical staff (doctors, lab techs) who may rotate passwords frequently, this creates a complete lock-out scenario that requires manual admin intervention. This is a table-stakes authentication feature that is entirely absent.
→ Add /forgot-password (email entry) and /reset-password/:token (new password entry) pages. Wire to /auth/forgot-password and /auth/reset-password backend endpoints.
🚫 Undo / Rollback for Destructive Actions
Clinical data deletion (PatientDetail), patient deletion (if implemented), and staff suspension are all permanent and immediate. The ISO 9241-110 Controllability principle requires that users be able to reverse actions. In a clinical context, accidental deletion of a vital signs record could have patient safety implications.
→ Implement a 10-second undo toast for clinical data deletion: "Record deleted. Undo". This uses optimistic UI deletion on the frontend with a deferred API call that can be cancelled. No new backend endpoints required.
⚠️ No ARIA Live Regions for Dynamic Content
Toast notifications, alert count updates, and loading state changes are purely visual. Screen readers have no way to announce "3 new unread alerts" or "Prediction completed successfully" without ARIA live region markup (aria-live="polite").
→ Wrap the toast container with aria-live="polite" aria-atomic="true". Add aria-live="polite" to the alert count badge in the header. The shadcn Toaster component can be extended to include this.
⚠️ No Contextual Help or Tooltips for Clinical Metrics
Clinical metrics (Risk Score, Confidence, SpO2, BP Systolic) are displayed with no contextual definition. A new doctor joining the platform has no in-app guidance on what a 78% risk score means clinically, or how to interpret the XAI feature contributions. This violates Bastien & Scapin's "Guidance" criterion entirely.
→ Add <Tooltip> components (already in the UI library) with clinical definitions for: Risk Score, Confidence, each vital sign abbreviation, and XAI contribution direction. Provide a link to a "How to read this report" help article.
⚠️ No Colour-Independent Risk Encoding
Across all risk badges (CRITICAL / HIGH / MODERATE / LOW), colour is the sole differentiator. Approximately 8% of male users have colour-blindness. For a medical decision-support tool, relying solely on colour for clinical severity classification violates WCAG 1.4.1 (Use of Color) and could lead to misinterpretation of patient risk.
→ Add redundant encoding: unique icons per risk level, text labels (never omitted), and distinct shapes where space allows. Example: CRITICAL = ▲ filled triangle (red) + bold text; LOW = ✓ circle (green) + regular text.
⚠️ No Onboarding or First-Run Experience
New organizations that complete registration arrive at a blank dashboard with no patients, no predictions, and no guidance. There are empty-state CTAs, but no structured onboarding: no setup checklist, no guided first prediction, no video walkthrough link. First-run abandonment is highest in clinical software where the learning curve is steep.
→ Add a dismissible onboarding banner on first login: "Welcome to DiagInfect. Get started: 1. Register your first patient → 2. Add clinical vitals → 3. Run your first AI prediction." Each step links directly to the relevant page. Mark complete in localStorage per user.
⚠️ No Profile Edit Capability
The Doctor Profile page (Profile.tsx) is entirely read-only. Doctors cannot update their username, email, preferred language, or set a profile photo. The Manager Profile has the same limitation. This violates the ISO 9241-11 user satisfaction principle — users expect to manage their own identity.
→ Add an "Edit" button that converts the profile fields to an editable form (inline edit or a dialog). At minimum, allow username and preferred language updates. Email changes should trigger re-verification.
⚠️ Date Formatting Without Explicit Locale
toLocaleDateString() and toLocaleString() are called throughout the app (Dashboard, PatientDetail, PredictionHistory, etc.) without an explicit locale argument. In an Algerian clinical context, the date format could render as MM/DD/YYYY in one browser and DD/MM/YYYY in another — a clinically dangerous ambiguity for birth dates and test timestamps.
→ Define a single formatDate(dateStr: string) utility in src/lib/utils.ts that uses new Intl.DateTimeFormat('fr-DZ', { day:'2-digit', month:'short', year:'numeric' }).format(new Date(dateStr)). Replace all raw toLocaleDateString() calls with this utility.
Section 3

UI/UX Flow & Friction Analysis — Categorized by Severity

CRITICAL CRITICAL FLOW FLAWS — Task Abandonment & User Disorientation
Alert Notification Leads Nowhere DoctorLayout.tsx
The primary notification entry point in the application — the bell icon with a badge counter — navigates to /doctor/dashboard instead of an alerts view. A doctor on the Patients page who sees "5 unread alerts" clicks the bell, lands on the same dashboard, and must visually scan the entire page to locate the alerts panel at the bottom right. This is a primary flow breakdown: the notification signal and its resolution are disconnected.
Redesign Recommendation
Create a dedicated /doctor/alerts page listing all alerts with filters (All / Unread / CRITICAL). Wire the bell icon to this page. The dashboard can continue showing a preview of 3–5 alerts with a "View all" link. This standard pattern (used by every major SaaS platform) resolves the disconnect entirely.
Lab Order Dual-CTA Causes Workflow Paralysis LabOrders.tsx
In the Lab Orders table, a PENDING order displays both a "Start" button AND an "Enter Results" button simultaneously. These are sequential actions in a defined workflow, not parallel alternatives. A new lab technician faced with two competing action buttons on a PENDING order has no way to know which to click first without external guidance. Clicking "Enter Results" on a PENDING order without starting it may create a workflow state error.
Redesign Recommendation
Enforce a strict state machine in the UI: PENDING → "Start" only. INPROGRESS → "Enter Results" only. Optionally, make "Start" automatically navigate to the results entry page so the two steps are merged into one action for the technician. Label the merged button: "Start & Enter Results."
Developer Annotation Displayed to Clinical Users NewPrediction.tsx
The subtitle of the New Prediction page reads verbatim: "Image 2 — runs Random Forest / XGBoost / NN on clinical + lab data." This is a code review annotation that was never removed. A clinician seeing this text would reasonably question the maturity and reliability of the platform, creating an immediate trust failure at the most critical moment in the user journey — the moment they're about to rely on AI for a patient decision.
Redesign Recommendation
Immediately remove and replace with: "Runs an ensemble AI model on the patient's latest clinical record and lab results." Conduct a full codebase search for similar inline annotations: grep for "Image [0-9]" in all .tsx files. This is a production bug.
HIGH HIGH FLOW FLAWS — Cognitive Overload & Repetitive Manual Input
Lab Results Entry — 8–15 Manual Row Entries Per Test EnterResults.tsx
For a complete CBC, a lab technician must: click "Add Analyte" 8+ times, type each analyte name from memory, enter the value, type the reference range (low and high), optionally set a panel group, and then select or auto-compute a flag. For a comprehensive metabolic panel this could be 20+ rows. No templates, no auto-fill from the test type, no import. This is the highest-friction task in the app.
Redesign Recommendation
Implement test templates: when the order has a recognized test type (CBC, BMP, LFT, Urinalysis), pre-populate rows with the standard analytes, their default reference ranges, and expected units. Show a "Load CBC Template" button when the test type matches. Allow managers to create custom templates. This single feature eliminates 80% of manual keystrokes for the lab role.
No Direct Patient→Prediction Shortcut from Detail Page PatientDetail.tsx / NewPrediction.tsx
Running a prediction for a specific patient requires: navigating to Patient Detail → clicking "Run AI Prediction" → being taken to the New Prediction page WITH the patient pre-selected (this part works via ?patient_id= query param). However, after a prediction is run, clicking "New Prediction" clears the patient selection entirely rather than defaulting to the same patient. A doctor running multiple iterations for the same patient must re-select them every time.
Redesign Recommendation
After a successful prediction result, change the "New Prediction" button label to "Run Again for [Patient Name]" and pass the same patient_id query parameter. Only offer a blank "New Prediction (different patient)" as a secondary option.
Staff Suspension — No Confirmation, No Reason, No Audit Trail Staff.tsx
A manager selecting "Suspend" from a staff dropdown instantly patches /manager/staff/:id/status with no confirmation step. If the manager is using a touch device and accidentally scrolls the dropdown to "Suspend" instead of "Deactivate," a doctor's access is revoked immediately. In a hospital context, this could interrupt an active patient consultation.
Redesign Recommendation
Gate all INACTIVE and SUSPENDED status changes behind a confirmation dialog with the staff member's name prominently displayed. Add an optional reason textarea. For SUSPENDED, show a clear warning: "This user will immediately lose access to all platform features." Deliver a success toast that includes an "Undo" button for 15 seconds post-action.
Clinical Symptoms Entry — Free-Text Comma-Separated PatientDetail.tsx
Symptoms are entered as a single text string, split by commas on save: symptoms.split(',').map(s => s.trim()). This creates three problems: (1) Inconsistent casing ("Fever" vs "fever") creates duplicate values in the backend. (2) Multi-word symptoms with commas fail (e.g., "nausea, vomiting, and diarrhea" becomes 3 separate entries including "and diarrhea"). (3) Doctors cannot search or filter by symptom across patients.
Redesign Recommendation
Replace with a tag-input component (e.g., a controlled combobox that adds chips on Enter/comma). Source suggestions from a curated 100-symptom list normalized to lowercase. Stored symptoms become a proper array of clean strings on every save. This also enables future "patients with this symptom" cross-queries.
Model Version Selection — Meaningless to Clinical Users NewPrediction.tsx
Doctors are asked to choose between model versions v2.3.1, v2.2.0, and v2.1.5. These version numbers carry no clinical meaning. No changelog, no accuracy comparison, no clinical validation notes are provided. In practice, all doctors will select v2.3.1 (the default), making the selector pure visual noise. It also creates a medico-legal ambiguity: if a prediction is acted upon on v2.2.0, who is responsible for not using the latest version?
Redesign Recommendation
Remove the model version selector from the doctor-facing UI entirely. Auto-select the latest model. If version auditability is required (e.g., for clinical validation), display the version used in the result card as a read-only reference: "Prediction made with Model v2.3.1." Expose version selection only in manager settings for institutional governance.

Duplicate API Calls on Every Layout Mount DoctorLayout.tsx + DoctorSidebar.tsx
Both DoctorLayout and its child DoctorSidebar fetch /doctor/alerts independently with separate useState + useEffect hooks. This fires 2 identical HTTP requests on every page load. The two components maintain separate unread counts that can drift apart if one updates before the other.
Redesign Recommendation
Consolidate into a single useAlerts() custom hook or a React Context at the DoctorLayout level. Pass the unread count down as a prop to DoctorSidebar. This eliminates one redundant network request per page navigation.
Inconsistent Date Formatting Across Locales All pages using toLocaleDateString()
At least 15 instances of raw toLocaleDateString() / toLocaleString() without locale arguments exist across Dashboard, PatientDetail, PredictionHistory, LabOrders, Staff, and Subscription pages. In an Algerian hospital setting (fr-DZ), dates can render ambiguously as MM/DD or DD/MM depending on the user's browser language setting.
Redesign Recommendation
Create a fmtDate(d: string) utility in src/lib/utils.ts using explicit Intl.DateTimeFormat with 'fr-DZ' locale. Global find-replace all new Date(...).toLocaleDateString() and .toLocaleString() calls in page files with this utility.
Profile Page is Fully Read-Only — No Edit Path src/pages/doctor/Profile.tsx + manager/Profile.tsx
Both the Doctor and Manager profile pages display user information (username, email, organization) with no way to edit any field. The preferred_lang field is fetched but never surfaced in the UI at all. Users who need to change their email or language preference have no in-app path to do so.
Redesign Recommendation
Add an "Edit Profile" button that transitions the card into an editable form. Minimally: username (editable), preferred_lang (select dropdown), email (editable with re-verification notice). Password change should be a separate "Change Password" section with current + new + confirm fields.
UpgradePrompt "Maybe Later" Has No Suppression Mechanism components/auth/UpgradePrompt.tsx
The upgrade prompt appears whenever a 402 error is thrown. "Maybe Later" dismisses it for the current interaction but it reappears on the next prediction attempt. There is no "Don't show again for today" or "Remind me in 3 days" option. For a doctor who knowingly operates on a trial plan, this is interruption noise on every prediction.
Redesign Recommendation
Add a "Remind me tomorrow" option that sets a localStorage flag with a timestamp. Suppress the full modal for 24 hours; instead show a dismissible banner: "Your trial limit has been reached. Upgrade." The full modal should still appear once per day as a gentle reminder.
XAI Contribution Bars — Inaccessible and Near-Invisible PredictionHistory.tsx (Sheet)
Feature contribution bars in the XAI side sheet are h-1.5 (6px tall) and have no ARIA attributes, no accessible text, no percentage label shown without hover. They are functionally unusable for screen reader users and barely readable for users with moderate visual impairments. The percentage is shown as a separate span but positioned to the right, not associated with the bar via ARIA.
Redesign Recommendation
Increase to h-3 minimum. Add role="meter", aria-valuetext with the formatted percentage. Pair the bar visually with the percentage value directly below it rather than in a separate column. Patients Page — No Sorting or Export Patients.tsx
A doctor with 150 patients and no sort controls must scroll through the entire grid to find recently added patients or patients with the highest risk scores. Redesign Recommendation
Add: (1) A sort dropdown: "Name A–Z | Highest Risk | Recently Added | Oldest First." (2) A CSV export button that downloads patient name, age, gender, risk level, and last prediction date. Both are client-side operations on the existing patients state array — no backend changes required.
